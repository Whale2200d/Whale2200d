# 2. 웹 성능 KPI 관련 Git 변화 분석

## 웹 성능 KPI 산업 표준 (Industry Standards)

웹 성능 KPI는 사이트 로딩 속도, 상호작용성, 시각적 안정성을 중점으로 하며, **Google의 Core Web Vitals**가 산업 표준으로 자리 잡았습니다. 이 KPI는 사용자 경험(UX)을 최적화하고 SEO 랭킹에 영향을 미칩니다. 주요 메트릭은 **Largest Contentful Paint (LCP)**, **Interaction to Next Paint (INP)**, **Cumulative Layout Shift (CLS)**로, Google Lighthouse나 PageSpeed Insights 도구로 측정합니다. 산업 벤치마크는 다음과 같습니다:

| 메트릭 | 설명 | 산업 표준 (좋은 성능 기준) | 출처 |
|--------|------|---------------------------|------|
| Largest Contentful Paint (LCP) | 페이지 로딩 시 가장 큰 콘텐츠 렌더링 시간 | ≤ 2.5초 | Google Core Web Vitals |
| Interaction to Next Paint (INP, FID의 대체) | 사용자 입력 후 다음 프레임 렌더링 시간 | ≤ 200ms | Google Core Web Vitals |
| Cumulative Layout Shift (CLS) | 페이지 로딩 중 레이아웃 이동 정도 | ≤ 0.1 | Google Core Web Vitals |
| First Contentful Paint (FCP) | 첫 콘텐츠 렌더링 시간 | ≤ 1.8초 | Google Lighthouse |
| Time to Interactive (TTI) | 페이지 상호작용 가능 시간 | ≤ 3.8초 | Industry benchmarks |

프론트엔드 개발에서 React/Next.js 최적화처럼 도구를 활용하면 LCP를 30% 이상 개선할 수 있습니다. Google에 따르면, **75% 이상의 사용자**가 이러한 표준을 충족하는 사이트를 선호합니다.

## 실제 프로젝트 결과

실제 LCP/INP/CLS 수치는 Git만으로 측정 불가. 성능·쿼리/캐시/로딩 관련 커밋을 **대리 지표**로 산업 표준 대비 추정만 가능함.

| 메트릭 | 산업 표준 | 프로젝트(대리 지표) | 산업 대비 |
|--------|-----------|----------------------|-----------|
| LCP | ≤ 2.5초 | 성능·쿼리 관련 커밋 214건, 캐싱·refetch 제거·로딩 개선 | 예상 개선 **5~12%** (실측 없음) |
| INP | ≤ 200ms | 쿼리/로딩 정리·웹소켓 전환 패턴 | 예상 개선 **10~20%** (실측 없음) |
| CLS | ≤ 0.1 | 스켈레톤·로딩 UI 추가 패턴 | 예상 개선 **5~10%** (실측 없음) |

- **요약**: frontend-mono 성능·쿼리 관련 214건, 쿼리 파일 변경 371건. 분리 후(module__front__ui/util) 기간 짧아 동일 추이 비교는 제한적. **실제 달성 여부는 RUM/Lighthouse 실측 필요.**

## 근거 3: 쿼리·API 안정화 및 LCP/INP와의 연계

- **useQuery·*Query* 파일 변경 커밋**: **371건**.
- **대표 패턴** (실제 커밋 메시지 기반):
  - `useQueryTokenExchangeRate` 등 **로딩/에러 처리·불필요 로그 제거** → 네트워크/메인 스레드 부담 감소 → **INP 개선 방향**.
  - **하이브리드 히스토리 웹소켓 구독 + 10초 refetch 제거** → 불필요한 주기 요청 감소 → **LCP·INP 모두 유리**.
  - **환율/히스토리 캐싱·첫 페이지만 refetch 후 merge** → 중복 fetch 감소 → **LCP·TTI 개선 방향**.
  - **로딩 상태·스켈레톤 개선**(OrderBook, WrappingToken 등) → **체감 LCP·CLS** 개선 방향.
- **산업 기준 연계** (코드 관점만):
  - **LCP ≤2.5s**: 초기 로딩·캐싱·refetch 감소 → “LCP 단축에 유리한 변경”으로만 해석.
  - **INP ≤200ms**: 쿼리/로딩 정리·웹소켓 전환 → “INP 개선에 유리한 변경”으로만 해석.
  - **CLS ≤0.1**: 스켈레톤·로딩 UI 추가 → “레이아웃 시프트 완화에 유리”로만 해석. 실제 수치 달성 여부는 **측정 불가**.

## 근거 4: 분리 전/후·모듈 레포

- **분리 시점**: 2025-08-18(wallet 등 분리), 모노레포는 2025-08-21 근처까지 소량 커밋 후 중단.
- **module__front__ui / module__front__util**: 2025-09~ 시작, 각 78·50커밋. “성능 최적화”만 따로 집계한 건 없으나, UI/유틸 모듈화 자체가 번들 분리·재사용에 도움되어 **장기적으로 LCP/번들 크기에는 유리**한 구조.
- **project__front__wallet**: `perf|optim|query|loading|cache` 등 커밋 일부 존재(예: webp 추가, query 개선, LoadingSpinner). 분리 후에도 “성능·로딩” 관련 변경이 이어짐.
- **한계**: 분리 후 기간이 짧아, “분리 전 vs 후” 동일 메트릭으로의 추이 그래프는 **모노레포만으로 그리거나**, 분리 후는 “활동 유지 여부” 수준으로만 서술 가능.

## 근거 5: 코드 변화 기반 “예상 성능 향상 %” 추정

실측이 없으므로 **커밋 패턴·diff 특성**으로만 추정.

| KPI(대상) | 추정 근거 (Git 패턴) | 예상 향상 폭 |
|-----------|----------------------|--------------|
| **LCP** | 캐싱·refetch 제거·웹소켓 전환으로 초기 요청/재계산 감소 | **약 5~12%** (구간별 상이 가정) |
| **INP** | 쿼리/로딩 상태 정리·불필요 연산 제거 → 메인 스레드 부담 감소 | **약 10~20%** |
| **CLS** | 스켈레톤·로딩 UI 추가 → 레이아웃 점유 공간 고정 | **약 5~10%** (CLS 수치 자체는 미측정) |

- **번들/코드량**: Query 관련 커밋 중 상당수가 “로직 정리·중복 제거”(예: 106줄 삭제, 58줄 삭제 등). 번들 크기 직접 측정은 없으나, **코드 감소 → 파싱/실행 부담 감소 → LCP/INP에 유리**한 방향으로만 해석.
