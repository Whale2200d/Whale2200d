# 4. 코드 효율성 분석

## 코드 효율성 산업 표준 (Industry Standards)

코드 효율성 메트릭은 유지보수성, 복잡도, 결함률을 중점으로 하며, **SonarQube**나 **CISQ** 표준이 벤치마크를 제공합니다. 산업 표준은 **코드 품질 지수(MI) ≥ 65**, **결함 밀도 ≤ 1/KLOC**로 설정됩니다. 주요 벤치마크는 다음과 같습니다:

| 메트릭 | 설명 | 산업 표준 (좋은 성능 기준) | 출처 |
|--------|------|---------------------------|------|
| Cyclomatic Complexity | 코드 경로 복잡도 | ≤ 10 | SonarQube & CISQ |
| Maintainability Index (MI) | 유지보수성 지수 | ≥ 65 | SonarQube |
| Defect Density | 코드 단위당 결함 수 | ≤ 1/KLOC | CISQ & DORA |
| Code Churn | 코드 변경률 | ≤ 20% (주간) | Waydev & Jellyfish |
| Test Coverage | 테스트 커버리지 | ≥ 80% | SonarQube & CISQ |

DORA 연구에 따르면, 고성능 팀은 이러한 표준을 준수하며, 코드 품질을 유지하면 **MTTR(Mean Time to Recovery)**이 50% 단축됩니다.

---

## 실제 프로젝트 결과

CC·MI·Test Coverage는 Git만으로 직접 측정 불가. churn·리팩터/제거 비율·테스트 이력을 **대리 지표**로 산업 표준 대비 추정함.

| 메트릭 | 산업 표준 | 프로젝트(대리 지표) | 산업 대비 |
|--------|-----------|----------------------|-----------|
| Cyclomatic Complexity | ≤ 10 | 리팩터·분리·단순화 커밋 비율 | 추정 **60/100** (정적 분석 없음) |
| Maintainability Index | ≥ 65 | 리팩터 20.5%(모노)·27.3%(wallet), 모듈화·제거 패턴 | 추정 **62/100** (목표 근접) |
| Test Coverage | ≥ 80% | 테스트 파일 변경 356건, vitest 존재 | 추정 **50/100** (Coverage % 미측정) |
| Defect Density | ≤ 1/KLOC | 이슈 없음 → 생략, churn·리팩터로 품질 추정 | — |
| **종합 (SonarQube 스타일)** | — | churn 332만(모노)·71만(wallet), 리팩터 비율 20~27% | **58/100** (보통, 도구 도입 시 상향 가능) |

## 근거 1: 분석 대상 및 제약

- **대상**: `frontend-mono`(2024-03 ~ 2025-08), `project__front__wallet`(2025-08~). 분리 시점 **2025-08-18** 기준 전/후 구간.
- **메트릭**  
  - **목표**: CC ≤10, MI ≥65, Test Coverage ≥80%.  
  - **Git 한계**: CC·MI·Coverage는 정적 분석/테스트 실행 필요 → **직접 측정 불가**.  
  - **대안**: churn(추가+삭제 라인), 리팩터/모듈화/중복 제거 커밋 비율, 테스트 파일 변경 이력으로 **추정·추이**만 서술. Defect Density는 **생략**.

---

## 근거 2: Churn Rate 및 분리 전/후 복잡도·변경량 추이

- **frontend-mono**  
  - **분리 전**(~2025-08-18): 누적 **add 1,866,558 / del 1,461,079 → churn 3,327,637**.  
  - **전체**: add 약 220만, del 약 167만, **churn 약 387만**.
- **project__front__wallet**  
  - **전체**: add 480,735, del 228,946, **churn 709,681**.
- **해석**: 분리 전 모노레포에서 변경량이 집중되고, 분리 후 wallet 단일 레포에서도 약 71만 라인 churn → **복잡도·규모 변화**는 “변경 패턴(리팩터 비율)”로 완화 여부를 보는 것이 타당함.

## 근거 3: 리팩터·모듈화·중복 제거 커밋 (SonarQube 스타일 평가)

- **frontend-mono**: `refactor|리팩터|제거|분리|통일|모듈` 관련 **1,316건** / 전체 6,425건 → **약 20.5%**.
- **project__front__wallet**: 동일 키워드 **145건** / 531건 → **약 27.3%**.
- **대표 사례**  
  - core 패키지 독립 라이브러리화, 브릿지 분리 준비  
  - Select Token·ReceiveAmountSection·레이아웃(Flex→grid)·스크롤(onActivated) 리팩터  
  - HybridTitle·PaybackSettingCautionDialog·useQueryTokenChart·불필요 console/테스트 코드 제거  
  - manage-address → rewards 모듈 이전, 타입/상수 정리
- **멀티 서비스 환경·공통 모듈화**: route·컴포넌트·api·쿼리 분리로 **중복 제거·일관성 확보** 패턴 존재. 정확한 “40% 중복 제거” 수치는 Git만으로 불가하나, **구조 개선·제거 커밋 다수**로 “상당한 중복 제거·모듈화”가 있었던 것으로 추정 → SonarQube 스타일로 **유지보수성·중복 제거** 항목 **62/100** 부여.

## 근거 4: Cyclomatic Complexity·Maintainability Index·Test Coverage 추정

| 메트릭 | 산업 목표 | Git 기반 추정 | 추정 점수(100점) |
|--------|-----------|----------------|------------------|
| **Cyclomatic Complexity** | ≤10 | 정적 분석 없음. 리팩터·분리·단순화 커밋 비율로 “복잡도 완화 노력”만 반영 | **60** |
| **Maintainability Index** | ≥65 | 리팩터 20%대·모듈화·제거 패턴 → 유지보수성 “보통 이상” | **62** |
| **Test Coverage** | ≥80% | 테스트 파일 변경 커밋 356건, vitest 설정 존재. Coverage 수치 없음 | **50** (존재·변경만 반영) |
| **Defect Density** | ≤1/KLOC | 이슈 트래커 없음 → **생략**. churn·리팩터 비율로 품질 추정만 가능 | — |

- **Test 관련**: `*.test.ts`, `*.spec.ts`, `__tests__` 경로 변경 커밋 **356건**. 테스트 코드 존재·이력은 있으나 **Coverage %는 미측정**.

---

## 근거 5: Churn Rate·변경 패턴으로 본 코드 품질 추정

- **Churn rate**: 기간별 (추가+삭제) / 해당 기간 코드베이스 규모. 현재 “코드베이스 규모”를 시점별로 보지 않았으므로, **월별 churn 절대치**로 “변동 강도”만 해석.
- **변경 패턴**: 리팩터·제거 비율이 20~27%로 유지 → “기능 추가만이 아닌 정리·구조 개선”이 일정 비율로 수행됨 → **품질·유지보수성에 긍정적**으로 해석.

## 산업 표준 비교 및 SonarQube 스타일 점수 요약

| 항목 | 산업 기준 | Git 기반 추정 결과 | 추정 점수 |
|------|-----------|---------------------|-----------|
| Cyclomatic Complexity | ≤10 | 직접 측정 불가, 리팩터 비율로 완화 노력 반영 | **60/100** |
| Maintainability Index | ≥65 | 리팩터·모듈화·중복 제거 패턴 → “보통 이상” | **62/100** |
| Test Coverage | ≥80% | 테스트 이력 있음, Coverage % 없음 | **50/100** |
| Defect Density | ≤1/KLOC | 이슈 없음 → 생략, churn/리팩터로 품질 추정 | — |
| **종합 (코드 효율성)** | — | churn·리팩터·모듈화 종합 | **58/100** |

- **결론**: Git만으로는 CC·MI·Coverage 실측이 불가하므로, **churn rate·리팩터/모듈화/중복 제거 비율**로 코드 품질·효율성을 추정해 **58/100**으로 평가. 실제 SonarQube·ESLint·Coverage 도구 도입 시 점수·목표 달성 여부를 정량 관리할 수 있음.
