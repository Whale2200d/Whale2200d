### 4. DORA & 코드 품질 지표 비교

#### 4.1 Deployment Frequency (배포 빈도, DORA)

- **frontend-mono** 기준:
  - PR 머지 **629건**, 기간 **2024-03-21 ~ 2025-08-08** (약 505일).
  - **일평균 PR 머지**: 629 ÷ 505 ≈ **1.25회/일** (전체 기간).
  - PR 머지가 발생한 날만 보면 **66일**에 629건 → 해당 날짜 기준 **일평균 약 9.5회** (최대 **33회/일**).
- 프로젝트 전체 커밋:

  - 머지 비율 ~16% → **기능 브랜치 → 머지** 패턴이 안정적.

**DORA 기준과 비교**

| DORA 기준 | 본 프로젝트 (정량) | 평가 |
|-----------|---------------------|------|
| Elite: 일일 여러 번~온디맨드 | PR 머지 **일평균 1.25회**(전체), 머지 있는 날은 **일평균 9.5회** | **일일 배포 가능 빈도 충족** (Elite에 근접) |
| High: 주 1회~일 1회 | 동일 | **충족** |

- CI/CD가 PR 머지와 연동되어 있다면 **Deployment Frequency**는 **DORA Elite~High** 수준으로 해석 가능.

#### 4.2 Lead Time for Changes (리드 타임, DORA)

- **DORA 권장**: Lead Time for Changes **&lt; 1일** (commit → production).
- Git 히스토리만으로는 “커밋→배포” 구간을 직접 측정할 수 없어, **PR 머지 빈도**로 간접 추정:
  - PR 머지 **일평균 1.25회** 이상 → 머지가 곧 배포라 가정하면 **배포까지 리드타임이 1일 이내인 비율이 높을 수 있는** 구조.
- **리뷰 시간 단축 (약 30% 목표)**:
  - 2024-03-28~29 **reviewer 서버·자동 리뷰어·PR 길이 개선** 도입으로, 리뷰 대기 구간 단축이 기대되는 파이프라인 구축.
  - Git 메타데이터만으로 30% 단축을 정확히 산출할 수는 없으나, **자동 리뷰어·알림·빌드/프리뷰 자동화**가 “코드 푸시 → 리뷰 완료” 구간을 줄이는 방향으로 기여한 것으로 해석 가능.

#### 4.3 Change Failure Rate (변경 실패율, DORA)

- **proxy**: 커밋 메시지에 `revert`, `hotfix` 포함 건수.
  - **frontend-mono**: 160건 / 6,425커밋 → **약 2.49%**
  - **project__front__wallet**: 소수 (전체 549커밋 중)
  - **전체(두 repo 합산)**: 실패/결함 대응 proxy 53~160 수준, 전체 6,956커밋 → **약 0.76%~2.49%** (mono만 보면 2.49%).

\[
\text{CFR (frontend-mono proxy)} \approx \frac{160}{6425} \approx 2.49\% , \quad
\text{CFR (전체 proxy)} \approx 0.76\% \sim 2.5\%
\]

- **DORA 기준**: 0~15%
- **평가**: **기준(0~15%) 이내**이며, **변경 실패율은 양호한 수준**으로 해석 가능.

#### 4.4 MTTR (Mean Time to Restore, DORA)

- **DORA 권장**: 복구 시간 **&lt; 1시간**.
- Git만으로는 실제 장애 복구 시간을 측정할 수 없음. **revert/hotfix 커밋**이 “문제 인지 후 복구 커밋”의 proxy라면, 같은 날 여러 revert/hotfix가 있으면 상대적으로 빠른 대응으로 해석 가능. 정량적 MTTR은 **운영/모니터링 데이터**로 보완 필요.

#### 4.5 Code Churn (코드 효율성 관점)

- 기준: **Code Churn ≤ 20%** (장기 유지보수에 이상적)
- 실제:
  - `frontend-mono`: 45.5%
  - `project__front__wallet`: 32.8%
  - 전체: 43.2%

**평가**

- **장점**
  - 초반 대규모 개발/리팩터링 단계에서는 churn이 높게 나오는 것이 일반적이다.
  - 모듈화·아키텍처 재정립·도메인 설계 변경 등이 있었다면, 이 구간의 높은 churn은 **투자/정리 과정**으로도 해석 가능하다.
- **리스크**
  - 장기간 비슷한 수준이 유지되면, 설계 안정화가 늦고 재작업과 버그 리스크가 커질 수 있다.
  - 특히 `frontend-mono`의 45%대 churn은 **구조 안정화가 더디거나 실험적 개발이 반복된 흔적**일 수 있다.

### 7. 종합 평가 – 프로젝트 건강도

#### 7.1 강점

- **배포 빈도·DORA**  
  - PR 머지 **일평균 1.25회**(505일간 629건), 머지 있는 날은 **일평균 9.5회** → **Deployment Frequency**는 DORA **Elite~High** 수준. Lead Time &lt;1일 가능 구조, **CFR(proxy) 2.49%**로 0~15% 기준 충족.
- **리뷰 시간 단축**  
  - 2024-03 reviewer·자동 리뷰어·빌드/프리뷰 자동화 도입으로, “리뷰 대기 구간” 단축이 기대되는 파이프라인 구축. 30% 단축 여부는 PR/이슈 트래킹 데이터로 보완 시 정량화 가능.
- **높은 커밋 및 머지 빈도**  
  - 약 12~13 커밋/일, 16%대 머지 비율로, 기능 브랜치 중심의 **빠른 반복/통합 사이클**이 형성되어 있다.
- **Change Failure Rate / Defect Density (proxy) 모두 업계 권장 기준보다 상당히 우수**  
  - `revert`/`hotfix` 기반 proxy로 볼 때, **실패율과 결함 밀도 모두 매우 낮은 수준**이다.
- **이관 후 규모 추이**  
  - frontend-mono 파일 수 **628→3,807**(+506%), wallet 이관 후 **2,416→2,579**(+6.7%)로 안정적 성장.
- **마이그레이션 이후 churn 감소**  
  - 새 repo(`project__front__wallet`)에서 churn이 내려가는 추세는, 구조 안정화 및 품질 개선 방향으로 움직이는 신호로 해석할 수 있다.

#### 7.2 리스크 및 개선 포인트

- **전반적으로 높은 Code Churn (40%대)**

  - 특히 `frontend-mono`에서의 45%대 churn은, **설계/아키텍처 원칙을 더 명확히 정립하고 모듈 경계를 고정할 필요**를 시사한다.
- **권장 개선 방향**
  - 아키텍처 원칙 문서화 및 공유
  - 도메인 경계(Bounded Context) 명확화
  - 중복/유사 기능 모듈 통합 및 리팩터링 정책 수립
  - 불필요한 실험/재작성 방지용 가이드(예: RFC 프로세스) 도입
