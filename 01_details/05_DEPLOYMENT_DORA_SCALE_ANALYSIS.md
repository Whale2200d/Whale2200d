## 2. DORA 표준 대비 정량 지표

### 2.1 Deployment Frequency (배포 빈도)

- **PR 머지**를 “배포 단위”의 proxy로 사용 (실제 배포는 CI/CD 설정에 따름).
- **frontend-mono** 기준:
  - PR 머지 **629건**, 기간 **2024-03-21 ~ 2025-08-08** (약 505일).
  - **일평균 PR 머지**: 629 ÷ 505 ≈ **1.25회/일** (전체 기간).
  - PR 머지가 발생한 날만 보면 **66일**에 629건 → 해당 날짜 기준 **일평균 약 9.5회** (최대 **33회/일**).
- 프로젝트 전체: 약 85~90 커밋/주, 12~13 커밋/일, 머지 비율 ~16% → 기능 브랜치 → 머지 패턴이 안정적.

**DORA 기준과 비교**

| DORA 기준 | 본 프로젝트 (정량) | 평가 |
|-----------|---------------------|------|
| Elite: 일일 여러 번~온디맨드 | PR 머지 **일평균 1.25회**(전체), 머지 있는 날은 **일평균 9.5회** | **일일 배포 가능 빈도 충족** (Elite에 근접) |
| High: 주 1회~일 1회 | 동일 | **충족** |

→ CI/CD가 PR 머지와 연동되어 있다면 **Deployment Frequency**는 **DORA Elite~High** 수준으로 해석 가능.

### 2.2 Lead Time for Changes (리드 타임)

- **DORA 권장**: Lead Time for Changes **< 1일** (commit → production).
- Git 히스토리만으로는 “커밋→배포” 구간을 직접 측정할 수 없어, **PR 머지 빈도**로 간접 추정:
  - PR 머지 **일평균 1.25회** 이상 → 머지가 곧 배포라 가정하면 **배포까지 리드타임이 1일 이내인 비율이 높을 수 있는** 구조.
- **리뷰 시간 단축 (약 30% 목표)**:
  - 2024-03-28~29 **reviewer 서버·자동 리뷰어·PR 길이 개선** 도입으로, 리뷰 대기 구간 단축이 기대되는 파이프라인 구축.
  - Git 메타데이터만으로 30% 단축을 정확히 산출할 수는 없으나, **자동 리뷰어·알림·빌드/프리뷰 자동화**가 “코드 푸시 → 리뷰 완료” 구간을 줄이는 방향으로 기여한 것으로 해석 가능.

### 2.3 Change Failure Rate (변경 실패율)

- **proxy**: 커밋 메시지에 `revert`, `hotfix` 포함 건수.
  - **frontend-mono**: 160건 / 6,425커밋 → **약 2.49%**
  - **전체(두 repo 합산)**: 약 0.76%~2.49%.
- **DORA 기준**: 0~15%
- **평가**: **기준 이내**이며, 변경 실패율은 양호한 수준으로 해석 가능.

### 2.4 MTTR (Mean Time to Restore)

- **DORA 권장**: 복구 시간 **< 1시간**.
- Git만으로는 실제 장애 복구 시간을 측정할 수 없음. revert/hotfix 커밋이 “문제 인지 후 복구 커밋”의 proxy라면, 같은 날 여러 revert/hotfix가 있으면 상대적으로 빠른 대응으로 해석 가능. **정량적 MTTR은 운영/모니터링 데이터로 보완 필요.**

### 2.5 DORA 요약

| 지표 | DORA 권장 | 본 프로젝트 (정량) | 평가 |
|------|-----------|---------------------|------|
| Deployment Frequency | 일일~온디맨드 | PR 머지 **일평균 1.25회**, 머지 있는 날 **9.5회/일** | **Elite~High** |
| Lead Time | < 1일 | PR 머지 기반 <1일 가능 구조 | Git 기준 추정 충족 |
| Change Failure Rate | 0~15% | **2.49%** (frontend-mono) | **우수** |
| MTTR | < 1시간 | - | 운영 데이터로 보완 필요 |
