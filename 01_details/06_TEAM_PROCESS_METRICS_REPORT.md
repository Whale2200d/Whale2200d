  산업에서 말하는 20~50 **스토리 포인트/스프린트**와 동일 구간으로 해석할 때, 1 PR ≈ 1 포인트로 매핑.
- **Cycle Time**: 머지 커밋의 **브랜치 첫 커밋일 → 머지일**까지 일수. Git `log branch_tip --not base`로 브랜치 최초 커밋 시점 추출.
- **WIP**: Little's law  
  \(\text{WIP} = \text{처리량(PR/주)} \times \text{Cycle Time(주)}\)  
  로 추정. 산업 권장 **WIP ≤ 5~10**과 비교.
- **데이터 생성**: `scripts/team_process_metrics.py` 실행 시 `team_process_metrics_report.json` 갱신.  
  Burndown 스타일 시각화는 `team_process_burndown.html`에서 확인.

### 3. Velocity · Cycle Time · WIP vs 산업 표준

#### 3.1 Velocity (20~50 포인트)

- **측정**: frontend-mono PR 머지 629건을 스프린트(2주) 단위로 집계, 머지가 1건 이상인 스프린트만 평균.
- **결과**: **약 41.9 PR/스프린트** → 20~50 포인트 구간 **내**.
- **해석**: 스크럼/애자일에서 말하는 “스프린트당 20~50 포인트”와 동일 구간으로 볼 수 있으며, **산업 표준 대비 양호**.

#### 3.2 Cycle Time (&lt; 1주)

- **측정**: 머지 커밋의 2nd parent(브랜치 tip)에 대해 `git log branch_tip --not base`로 브랜치 최초 커밋일 추출, 머지일과의 차이(일) 계산.
- **결과**: 샘플 기준 **평균 0일**, **100%가 7일 미만** (동일일 머지 비율 높음).
- **해석**: Git만으로 측정한 **Cycle Time은 산업 목표 “&lt; 1주”를 충족**.  
  (실제 “작업 시작→배포” 리드 타임은 CI/배포 로그와 결합 시 더 정확히 측정 가능.)

#### 3.3 WIP (≤ 5~10)

- **추정**:  
  \(\text{WIP} \approx \frac{\text{PR 머지 수(505일)}}{505/7} \times \frac{\text{Cycle Time(일)}}{7}\)  
  Cycle Time이 0에 가까우면 산업 가정 3일 등으로 대체하여 계산.
- **결과**: 추정 WIP **≤ 10** (권장 상한 이내).
- **해석**: **동시에 진행 중인 작업 수(WIP)가 산업 권장(≤5~10) 범위**로 해석 가능.

### 7. 종합 평가 – 팀 프로세스 관점

#### 7.1 강점

- **Velocity·Cycle Time·WIP**  
  - Git 기준 **Velocity 41.9 PR/스프린트**(20~50 구간), **Cycle Time 100% &lt; 7일**, **WIP ≤ 10**으로,  
    산업에서 말하는 **Velocity 20~50 포인트, Cycle Time &lt; 1주, WIP ≤ 5~10**과 정합한다.
- **PR 파이프라인·알림·CI**  
  - PR 파이프라인/리뷰 자동화(2024-03), reviewer 알림·Git notify(2024-10~11), Job Queue(2025-01) 등  
    **프로세스 개선 커밋**이 이어졌고, 리뷰 대기 단축·배포/CI 가시성 확보에 기여한 것으로 해석 가능.
- **이관 반영 시각화**  
  - 스프린트/기간별 커밋 패턴을 **Burndown 스타일**로 정리하고, **이관 구간**을 명시해 두 repo 연속성을 볼 수 있다.

#### 7.2 해석 시 유의사항

- Velocity·Cycle Time·WIP는 **Git 로그만**으로 추정한 값이다.  
  Jira/이슈 트래커의 **스토리 포인트·실제 배포 이력**과 결합하면 더 정확해진다.
- **프로세스 개선 전·후 “효율 향상 %”**는 구간별 PR 밀도 차이로 인해 단일 수치로 해석하기 어렵다.  
  **산업 기준 대비 메트릭 충족 여부**로 평가하는 것이 타당하다.
- 이관 구간에서는 두 저장소 커밋이 공존하므로, **남은 작업량**은 두 repo 합산 기준으로 정의된다.

#### 7.3 사용 방법

1. **메트릭·JSON 재생성**  
   ```bash
   python3 scripts/team_process_metrics.py
   ```
2. **Burndown 차트**  
   `team_process_burndown.html`을 브라우저에서 연다 (같은 디렉터리에 `team_process_metrics_report.json` 필요).
